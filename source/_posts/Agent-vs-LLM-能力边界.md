# Agent ≠ LLM：聊聊 AI 编程工具背后的能力边界

最近和同事聊到 DeepSeek V4 Pro 发布，文档里写"Agent 能力大幅提升"。一位同事疑惑："Agent 能力不是靠工程提升的吗？它同时发布了类似 Cursor 或 Claude Code 的 IDE/CLI 工具？"

这是个很经典的混淆，也很值得拆开讲。如果你用过 Claude Code、Cursor、Cline 这类 AI 编程工具，又对它们背后的原理半懂不懂，这篇文章可能能帮你建立一个清晰的分层认知。

## 一、Agent = Harness × Model：两层能力，独立演化

AI Agent 系统由两层组成：

| 层 | 谁负责 | 例子 |
|---|---|---|
| Harness（外壳/工程） | 工程团队 | Claude Code、Cursor、Cline、Aider、Devin |
| Brain（大脑/模型） | 模型训练团队 | Opus、Sonnet、GPT-5、DeepSeek V4 |

**关键事实：把同一个 harness 配不同模型，效果差几倍。**

这就证明 harness 提供的是**舞台**，但能不能跳好舞，是**模型**的事。所以"模型的 Agent 能力"是个独立的、可训练、可评测的目标 —— 这跟模型团队有没有同时发 IDE 完全无关。

一个类比：

> Harness 像 F1 赛车，模型像车手。Red Bull 的车再好，给新手开也跑不出成绩；舒马赫的水平再高，给他一辆五菱也赢不了。
>
> "DeepSeek V4 Agent 能力提升" = "这个车手训练完，水平提升了"。这跟"他有没有自己的车队"是两件事。

## 二、工具调用：决策层和执行层的分工

很多人会想：工具调用不是 Agent 的能力吗？模型只负责生成文字，调工具是 Agent 的事啊。

这个直觉对了一半，但搞混了"谁决定"和"谁执行"。

**工具调用其实是两层：**

1. **决策层（模型做）**：什么时候该调工具、调哪个、参数怎么填、结果怎么解读、下一步怎么走 —— 全是模型输出的。模型必须在训练阶段就被教会"输出结构化的 tool_use 块"这种格式。
2. **执行层（Harness 做）**：把模型输出的 tool_use 真正跑起来（执行 bash、读文件、调 API），把结果塞回 context 给模型看。

Claude Code CLI 只是第 2 层 —— 一个调度器和沙箱。**真正"会用工具"的是模型本身**。同样是 Claude Code，套个不会工具调用的旧模型（比如 GPT-3 时代的），它就废了，连续调 3 步就乱套。

## 三、长时任务能力：context 管理是 Harness 的事，"用好 context"是模型的事

接下来是更深的一层困惑：每次模型推理都是无状态的，Harness 把整段 history 塞给模型而已。那"跨几十轮保持目标不漂移"不也是 Harness 的功劳吗？

**给了上下文 ≠ 能用好上下文。**

同样一段 50 轮的 history 塞给不同模型，结果天差地别：

- **弱模型**：注意力被最近几轮带偏，忘了 30 轮前用户定的约束，重复调用刚才失败过的工具，把工具返回的中间结果当成最终答案。
- **强模型**：能真的"读回去"找到原始目标，记得自己已经试过哪些路径，在错误堆积时主动回退或换方案。

这里涉及两个独立但相关的模型能力：

### Long-context utilization（长上下文有效利用）

**问题**：模型号称支持 200K token，但塞满了真的都"看见"了吗？

测的是**信息检索的空间维度** —— 在长文本任意位置埋一条关键信息，模型能不能找出来用对。经典测试是 **Needle in a Haystack（大海捞针）**：在 100K token 的小说中间塞一句"今天北京 28 度"，最后问天气。

典型失败模式叫 **"Lost in the middle"** —— 模型对开头结尾敏感，对中间内容视而不见。在 Agent 场景下，就是 history 里第 1 轮用户说"千万别动数据库"，到第 40 轮模型就忘了，照样去 drop table。

### Long-horizon coherence（长程任务连贯性）

**问题**：一个目标需要 50 步操作完成，模型能不能始终朝着这个目标走而不跑偏。

测的是**决策的时间维度**。经典测试是 **SWE-bench**：解决一个真实 GitHub issue，需要看代码、改代码、跑测试、改 bug、再跑测试…几十轮工具调用后还能不能完成原始任务。

典型失败：

- **目标漂移**：开始让它修 bug A，修着修着发现代码有点丑，跑去重构，忘了 bug A 还没修。
- **死循环**：tool 报错 → 重试 → 再报错 → 再重试，意识不到该换方案。
- **虚假完成**：自己说"已完成"，但其实啥也没改。
- **重复劳动**：不记得自己 10 轮前已经读过这个文件，又读一遍。

| | Long-context utilization | Long-horizon coherence |
|---|---|---|
| 维度 | 空间（context 内位置） | 时间（多轮决策） |
| 像啥 | 阅读理解 | 执行力 |
| 失败表现 | "看不见"早期信息 | "记得"但跑偏 |

**前者是后者的前提。** 这两个能力都是后训练阶段专门优化的，Harness 帮不了多少。Anthropic、OpenAI、DeepSeek 内卷的就是这部分。

## 四、一个有意思的反转：Compact 是谁做的？

Claude Code 有个 `/compact` 功能：context 太长时把历史压缩成摘要。这是 Agent 的能力还是模型的能力？

**两者协作，但分工和工具调用反过来。**

| 步骤 | 谁做 |
|---|---|
| 1. 触发（监测 token 用量） | Agent |
| 2. 摘要（理解 + 总结对话） | **模型** |
| 3. 替换（把摘要塞回 history） | Agent |

**核心认知工作（"什么重要、什么能扔"）是模型在做。** Agent 只是搬运工 + 决策何时触发。

对比工具调用：

| | 决策层 | 执行层 |
|---|---|---|
| 工具调用 | 模型（决定调啥） | Agent（去跑 bash） |
| Compact | Agent（决定何时压缩） | 模型（写摘要） |

工具调用是"模型动脑、Agent 动手"；Compact 是"Agent 动脑、模型动手"。**没有一个清晰的"模型只管说话、Agent 只管干活"的边界**，两者是深度耦合的协作关系。

## 五、Claude Code 教会我们的 Agent 工程原则

理解了分层之后，再看 Claude Code 的设计就能学到很多。这部分对自己想做 Agent 的人尤其有用。

### 1. 工具设计：小而锐 > 大而全

Claude Code 把 "读文件" 和 "编辑文件" 都拆成两个工具（Read / Edit）。

**启发**：工具就是 LLM 的 API。给一个 `do_anything(prompt)` 模型反而用不好，因为决策空间太大。**工具粒度越细，模型每一步选择越确定**。每个工具的 description 是 prompt engineering 的主战场 —— Claude Code 的 Bash 工具 description 几百行，专门教模型"什么时候别用 cat 用 Read"。

### 2. Context 当显存管，不当硬盘

Claude Code 的做法：分层加载（先读 index 再按需读细节）、子 Agent 开新 context（结果只回传摘要）、`/compact` 主动压缩。

**启发**：每塞一个 token 就有成本（钱、延迟、attention 稀释）。"懒加载"比"预加载"重要。**子 Agent 的本质是 context isolation** —— 让一个 fresh 模型去搜 50 个文件，主对话只看到 200 字总结。这是上下文窗口的"分页机制"。

### 3. 状态外置：让模型"看得见"自己干了啥

Claude Code 提供 TodoWrite 工具强制把计划写出来，提供 Memory 系统持久化跨会话事实。

**启发**：模型是无状态的，但**任务是有状态的**。差额必须落在外部介质上。把"心里想的"变成"屏幕上写的"，能显著降低跑偏概率 —— 因为下一轮模型读 history 时，会被自己上一轮写的 todo "锚定"。

这跟 GTD（Getting Things Done）方法论一个道理：**写下来 = 卸载认知负担**。人脑作为存储器很烂（容易忘、会焦虑），但作为处理器很强 —— 所以把待办全扔到外部系统里。LLM Agent 和 GTD 是同构的：都是给"健忘的处理器"配一个"持久的笔记本"。

### 4. 信任分层：别让模型决定一切

Claude Code 的安全防线从外到内：

1. **Hooks**（确定性脚本）：PreToolUse 拦截危险命令
2. **Permissions**（白名单）：settings.json 控制哪些命令免确认
3. **System prompt 约束**：CLAUDE.md 写死规则
4. **模型判断**：最后才轮到模型自己想

**启发**：**安全关键的事永远不靠 prompt 兜底**。"你不要 rm -rf" 写一万遍，遇到对抗性输入还是会破防。Hooks 是确定性代码，不可绕过。模型层、Agent 层、OS 层各司其职，不要试图把所有逻辑塞进 prompt。

### 5. Verification > Vibes：不能验证就明说

Claude Code 的 system prompt 里有一条规则：

> "If you can't test the UI, say so explicitly rather than claiming success."

字面意思：如果你没法测 UI，就明说，别声称成功。

背景是：用户让模型改一个前端按钮，模型改完代码、TypeScript 编译过了、单测过了，于是说"已完成 ✅"。但按钮在浏览器里其实是错位的。**编译通过 ≠ 功能正确**。

这条规则反对的是"虚假完成"（false completion）—— 模型最容易翻车的失败模式之一。模型有强烈的"取悦用户"倾向，会把"我尽力了"包装成"我搞定了"。规则强制它**把验证边界讲清楚**：哪些验证过了、哪些没法验证、用户需要自己补哪部分。

更普适的原则：**用确定性证据替代模糊感觉**。Agent 工程的可靠性提升，本质都在做这件事。

### 6. Skills / Slash / Hooks：让用户能编程 Agent

Claude Code 把"自定义工作流"做成一等公民 —— 用户写一个 markdown 文件就是一个 skill，不需要改源码。约定优于配置：CLAUDE.md 就是一个文件，没有什么神秘 schema。

**启发**：Agent 不该是封闭系统。这种开放性带来生态爆炸，是 Agent 平台化的关键。

### 7. 协作分寸：克制是高级能力

Claude Code 的 system prompt 里相当大篇幅在教**什么时候说话、说多长**：

- 探索性问题用 2-3 句回答
- 任务结束的 summary 一两句话
- 不要 narrate 思考过程
- 不要顺手重构、不要加不必要的注释、不要写没人要的文档

**启发**：模型默认行为是话痨 + 过度发挥，必须显式压制。Agent 行为最难调的不是能力，是分寸 —— 什么时候问、什么时候直接做、回话多长。

## 六、Claude Code 的 system prompt 有多卷

最后一个有意思的事实：Claude Code 一次会话的 system prompt + 项目文档加起来，**保守估计 8000~15000 token**，相当于一篇硕士论文的引言。

对比早期 GPT 调用的 system prompt 一般几十到几百 token，这是**两个数量级以上的投入**。Anthropic 愿意付这个钱，是因为：

1. **省的更多**：少一次错误修正循环就回本（一次错误的 git push reset 可能要 10 轮 token 来救）
2. **比微调便宜**：用 prompt 修行为，今晚改完明天就生效，不用重训模型
3. **可解释**：行为不对，能直接定位到 prompt 哪一行漏了
4. **可分发**：CLAUDE.md / AGENTS.md 让用户也能塞自己的规则

每一条规则背后都是**真实事故的沉淀**。如果你看到一条奇怪的规则，大概率背后有个具体的失败 case。

## 七、给自己想做 Agent 的人的 7 条提炼

1. **工具是 SDK，不是函数列表** —— 认真写 description，认真拆粒度
2. **Context window 当显存管** —— 索引、摘要、子 Agent 三件套
3. **状态外置 + todo 强制写出** —— 防漂移最便宜的手段
4. **安全靠 Hooks，不靠 prompt** —— 模型是说服对象，不是裁判
5. **作者 / 审稿分开** —— 同 context 自审约等于零
6. **让用户能扩展** —— Skills + 配置文件 + Hooks 三层开放
7. **教模型"什么时候别动手"** —— 比"什么时候动手"更难也更重要

## 收尾：回到那个困惑

回到开头同事的疑问："Agent 能力不是靠工程提升的吗？"

正确答案是：

> **Agent 能力 = 模型能力 × 工程能力。两者都重要，但是是两层独立的事。**
>
> 模型团队提升的是"装在别人 Harness 里也能跑得好"的那部分能力 —— 工具使用准确性、长程目标保持、错误恢复、诚实边界。这些通过 SWE-bench、TAU-bench 这类标准评测衡量，**评测时所有模型用的是同一个 Harness，唯一变量是模型**。
>
> 所以 DeepSeek V4 没出 CLI 不是矛盾 —— 它要提升的是模型那一层，跟 Harness 是不是自家的无关。
